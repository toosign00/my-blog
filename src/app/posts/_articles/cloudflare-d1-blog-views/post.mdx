블로그를 거의 다 만들고 나서 홈 화면 우측 카드에 넣을 콘텐츠를 고민했습니다. 뭐가 있을까 하다가 마침 떠오른 게 조회수였는데, 포스트 카드에 숫자가 보이면 소소하게 읽을 맛도 날 것 같았습니다.

Vercel Analytics를 이미 붙여뒀지만, 그쪽은 대시보드에서만 확인할 수 있고 페이지나 게시물에 직접 숫자를 노출하려면 직접 집계해서 저장하는 DB가 따로 필요했습니다. Vercel에 배포 중인 Next.js 블로그라 Supabase가 설정은 더 간단하지만, 도메인, S3 스토리지, 이메일 인증까지 이미 Cloudflare로 관리하고 있던 터라 DB도 같은 생태계 안에서 해결하고 싶었습니다.

---

## Cloudflare D1이 뭔가요?

D1은 Cloudflare가 제공하는 서버리스 SQLite 데이터베이스입니다. Cloudflare Workers와 함께 쓰도록 설계됐지만, REST API도 제공하기 때문에 Next.js 같은 외부 환경에서도 HTTP 요청으로 쿼리를 날릴 수 있습니다.

이 블로그는 Vercel에 배포되어 있어 Workers 바인딩을 쓸 수 없는 상황이었는데, REST API 덕분에 문제없이 연결할 수 있었습니다. 선택한 이유는 단순했습니다. 무료 플랜으로 충분하고, SQL을 그대로 쓸 수 있고, 별도 인프라 관리가 필요 없었습니다.

---

## 테이블 구조

조회수를 날짜별로 집계하는 단순한 구조입니다.

```sql
CREATE TABLE page_views (
  id       INTEGER PRIMARY KEY AUTOINCREMENT,
  date     TEXT    NOT NULL,
  pathname TEXT    NOT NULL,
  count    INTEGER NOT NULL DEFAULT 0,
  UNIQUE(date, pathname)
);
```

`date`와 `pathname`을 복합 유니크 키로 잡아서, 같은 날 같은 페이지의 조회수를 한 행에 누적합니다.

---

## D1에 쿼리 날리기

Cloudflare REST API를 직접 호출하는 유틸 함수를 하나 만들었습니다.

```ts
export async function queryD1<T = Record<string, unknown>>(
  sql: string,
  params: (string | number)[] = []
): Promise<T[]> {
  const accountId = process.env.CLOUDFLARE_ACCOUNT_ID;
  const databaseId = process.env.CLOUDFLARE_D1_DATABASE_ID;
  const apiToken = process.env.CLOUDFLARE_API_TOKEN;

  const res = await fetch(
    `https://api.cloudflare.com/client/v4/accounts/${accountId}/d1/database/${databaseId}/query`,
    {
      method: 'POST',
      headers: {
        Authorization: `Bearer ${apiToken}`,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({ sql, params }),
    }
  );

  const data = await res.json();
  return data.result?.[0]?.results ?? [];
}
```

환경변수 3개(`CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_D1_DATABASE_ID`, `CLOUDFLARE_API_TOKEN`)만 있으면 어디서든 SQL을 실행할 수 있습니다.

## 조회수 기록 API

Next.js Route Handler로 `GET`(조회)과 `POST`(기록) 두 가지를 만들었습니다.

조회수를 기록할 때 한 가지 문제가 있었습니다. 새로고침할 때마다 카운트가 올라가면 안 된다는 것입니다. **쿠키로 마지막 카운트 시각을 기록해서 15분 이내 재방문은 카운트하지 않도록** 처리했습니다.

```ts
const COUNT_INTERVAL_MS = 15 * 60 * 1000;

// 15분 이내 재방문이면 카운트 스킵
if (lastCountedAt > 0 && now - lastCountedAt < COUNT_INTERVAL_MS) {
  return NextResponse.json({ ok: true, counted: false });
}
```

카운트가 발생했을 때는 `INSERT ... ON CONFLICT DO UPDATE`로 날짜별 행을 upsert합니다.

```sql
INSERT INTO page_views (date, pathname, count)
VALUES (?, ?, 1)
ON CONFLICT(date, pathname) DO UPDATE SET count = count + 1
```

## 오늘 조회수 / 전체 조회수 계산

날짜별로 저장해뒀기 때문에 오늘 조회수와 누적 조회수를 한 번의 쿼리로 뽑을 수 있습니다.

```sql
SELECT
  COALESCE(SUM(CASE WHEN date = ? THEN count ELSE 0 END), 0) AS today,
  COALESCE(SUM(count), 0) AS total
FROM page_views
WHERE pathname = ?
```

`date = ?` 조건이 맞는 행만 `today`로, 전체 합산은 `total`로 가져옵니다.

## TanStack Query로 실시간 반영

처음에는 서버 컴포넌트에서 조회수를 가져왔는데, 문제가 있었습니다. 페이지가 렌더링된 시점의 조회수가 보이기 때문에, 지금 막 올라간 내 조회수는 화면에 빠져 있었습니다.

TanStack Query를 붙여서 해결했습니다. `POST /api/stats`의 응답에 `counted: true`가 오면 쿼리를 invalidate해서 최신 조회수를 다시 fetch합니다.

```ts
const mutation = useMutation({
  mutationFn: () => postStats(pathname),
  onSuccess: (data) => {
    if (data.counted) {
      queryClient.invalidateQueries({ queryKey: ['stats', pathname] });
    }
  },
});
```

이렇게 하면 내가 방문한 순간 조회수가 올라가는 모습을 바로 확인할 수 있습니다.

## 마치며

D1은 간단한 집계 용도로 쓰기에 딱 맞습니다. Workers 없이 REST API만으로도 충분히 연결할 수 있고, 무료 플랜에서도 개인 블로그 트래픽 정도는 여유롭게 처리됩니다.

조회수처럼 가벼운 기능에 별도 DB 서비스를 붙이는 게 과하다고 느껴질 수 있는데, D1은 설정 비용이 거의 없어서 부담이 없었습니다.
