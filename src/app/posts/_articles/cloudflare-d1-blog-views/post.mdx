블로그를 거의 다 만들고 나서 홈 화면 우측 카드에 넣을 콘텐츠를 고민했습니다. 뭐가 있을까 하다가 마침 떠오른 게 조회수였는데, 포스트 카드에 숫자가 보이면 소소하게 읽을 맛도 날 것 같았습니다.

Vercel Analytics를 이미 붙여뒀지만, 그쪽은 대시보드에서만 확인할 수 있고 페이지나 게시물에 직접 숫자를 노출하려면 직접 집계해서 저장하는 DB가 따로 필요했습니다. Vercel에 배포 중인 Next.js 블로그라 Supabase가 설정은 더 간단하지만, 도메인, S3 스토리지, 이메일 인증까지 이미 Cloudflare로 관리하고 있던 터라 DB도 같은 생태계 안에서 해결하고 싶었습니다.

---

## Cloudflare D1이 뭔가요?

D1은 Cloudflare가 제공하는 서버리스 SQLite 데이터베이스입니다. Cloudflare Workers와 함께 쓰도록 설계됐지만, REST API도 제공하기 때문에 Next.js 같은 외부 환경에서도 HTTP 요청으로 쿼리를 날릴 수 있습니다.

이 블로그는 Vercel에 배포되어 있어 Workers 바인딩을 쓸 수 없는 상황이었는데, REST API 덕분에 문제없이 연결할 수 있었습니다. 선택한 이유는 단순했습니다. 무료 플랜으로 충분하고, SQL을 그대로 쓸 수 있고, 별도 인프라 관리가 필요 없었습니다.

---

## 테이블 구조

방문 이벤트를 행 단위로 기록하는 구조입니다.

```sql
CREATE TABLE page_views (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  ip         TEXT    NOT NULL,
  pathname   TEXT    NOT NULL,
  visited_at INTEGER NOT NULL
);
```

`ip`와 `visited_at`을 함께 저장해서, 짧은 시간 안에 같은 IP가 같은 페이지를 재방문하면 카운트하지 않도록 서버에서 직접 판단합니다.

---

## D1에 쿼리 날리기

Cloudflare REST API를 직접 호출하는 유틸 함수를 하나 만들었습니다.

```ts
export async function queryD1<T = Record<string, unknown>>(
  sql: string,
  params: (string | number)[] = []
): Promise<T[]> {
  const accountId = process.env.CLOUDFLARE_ACCOUNT_ID;
  const databaseId = process.env.CLOUDFLARE_D1_DATABASE_ID;
  const apiToken = process.env.CLOUDFLARE_API_TOKEN;

  const res = await fetch(
    `https://api.cloudflare.com/client/v4/accounts/${accountId}/d1/database/${databaseId}/query`,
    {
      method: 'POST',
      headers: {
        Authorization: `Bearer ${apiToken}`,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({ sql, params }),
    }
  );

  const data = await res.json();
  return data.result?.[0]?.results ?? [];
}
```

환경변수 3개(`CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_D1_DATABASE_ID`, `CLOUDFLARE_API_TOKEN`)만 있으면 어디서든 SQL을 실행할 수 있습니다.

## 조회수 기록 API

Next.js Route Handler로 `GET /api/views`(조회)와 `POST /api/views`(기록) 두 가지를 만들었습니다.

새로고침할 때마다 카운트가 올라가면 안 되기 때문에, **쿠키 대신 IP 기반으로 중복 방문을 서버에서 걸러냅니다.** 로컬호스트 IP는 즉시 스킵하고, 같은 IP가 30분 이내에 같은 페이지를 재방문한 기록이 있으면 카운트하지 않습니다.

```ts
const INTERVAL_MS = 30 * 60 * 1000;

const ip = (request.headers.get('x-forwarded-for') ?? '127.0.0.1').split(',')[0].trim();

if (ip === '127.0.0.1' || ip === '::1') {
  return NextResponse.json({ ok: true, counted: false });
}

const cutoff = Math.floor((Date.now() - INTERVAL_MS) / 1000);
const recent = await queryD1<{ cnt: number }>(
  `SELECT COUNT(*) as cnt FROM page_views WHERE ip = ? AND pathname = ? AND visited_at >= ?`,
  [ip, pathname, cutoff]
);

if ((recent[0]?.cnt ?? 0) > 0) {
  return NextResponse.json({ ok: true, counted: false });
}
```

중복이 아니면 현재 Unix 타임스탬프와 함께 행을 insert합니다.

```ts
const now = Math.floor(Date.now() / 1000);
await queryD1(`INSERT INTO page_views (ip, pathname, visited_at) VALUES (?, ?, ?)`, [
  ip,
  pathname,
  now,
]);
```

## 오늘 조회수 / 전체 조회수 계산

행 단위로 저장하기 때문에 오늘 조회수와 누적 조회수를 한 번의 쿼리로 뽑을 수 있습니다. `visited_at`이 Unix 타임스탬프라서, 오늘 자정(Asia/Seoul 기준)을 기준으로 조건을 나눕니다.

```sql
SELECT
  COALESCE(SUM(CASE WHEN visited_at >= ? THEN 1 ELSE 0 END), 0) AS today,
  COUNT(*) AS total
FROM page_views
WHERE pathname = ?
```

`?`에는 오늘 자정의 Unix 타임스탬프를 넘깁니다. `visited_at >= ?`에 해당하는 행만 `today`로, 전체 행 수는 `total`로 가져옵니다.

## TanStack Query로 실시간 반영

처음에는 서버 컴포넌트에서 조회수를 가져왔는데, 문제가 있었습니다. 페이지가 렌더링된 시점의 조회수가 보이기 때문에, 지금 막 올라간 내 조회수는 화면에 빠져 있었습니다.

TanStack Query를 붙여서 해결했습니다. `POST /api/views`의 응답에 `counted: true`가 오면 쿼리를 invalidate해서 최신 조회수를 다시 fetch합니다.

```ts
const mutation = useMutation({
  mutationFn: () => postViews(pathname),
  onSuccess: (data) => {
    if (data.counted) {
      queryClient.invalidateQueries({ queryKey: ['views', pathname] });
    }
  },
});
```

이렇게 하면 내가 방문한 순간 조회수가 올라가는 모습을 바로 확인할 수 있습니다.

## 마치며

D1은 간단한 집계 용도로 쓰기에 딱 맞습니다. Workers 없이 REST API만으로도 충분히 연결할 수 있고, 무료 플랜에서도 개인 블로그 트래픽 정도는 여유롭게 처리됩니다.

조회수처럼 가벼운 기능에 별도 DB 서비스를 붙이는 게 과하다고 느껴질 수 있는데, D1은 설정 비용이 거의 없어서 부담이 없었습니다.
